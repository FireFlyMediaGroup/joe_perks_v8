Section 1 — Design Philosophy & Architecture

Core design principle: two paths, one experience
The roaster fulfillment system has two distinct entry points — email (magic link) and portal (authenticated dashboard) — but they must feel like the same product. A roaster who primarily uses email should feel the portal is familiar when they finally visit it. A roaster who lives in the portal should get emails that feel like a lightweight version of the dashboard.
This is achieved through a single shared OrderFulfillmentSheet component (packages/ui) that renders in both contexts. The magic link page IS the minimal portal — not a different product.

Apple Human Interface Guidelines — applied to a web product
Apple HIG principles translate directly to web design patterns used throughout this feature:

HIG Principle
Applied as
Clarity — content over chrome
Magic link page strips all navigation. Only the task matters: ship this order.
Deference — UI recedes, content leads
The order details and payout headline dominate. The form is secondary until the roaster is ready for it.
Depth — layered information
Payout breakdown collapsed behind a toggle. Event timeline collapsed behind a disclosure. Full financial detail in the ledger. Information revealed progressively.
Minimum 44pt touch targets
All buttons, carrier pills, tab bar items, and action links are 44px minimum height/width.
Feedback within 100ms
Carrier selection highlights immediately. Submit shows a spinner before the network round-trip returns.
Focused task screens
The magic link page, escalation sheet, and tracking edit form each have one primary action and no distracting navigation.
Confirmation for destructive actions
Can't fulfill requires a two-step sheet. Edit tracking requires an inline confirm. Auto-refund explained clearly before it happens.
Color carries meaning, never alone
SLA urgency uses color AND icon. Status pills use color AND text label. Never rely on color alone (accessibility).

Fulfillment flow architecture

Two fulfillment paths — one shared component
PATH A — Email first (MVP default): Roaster receives email → clicks "Fulfill this order" → lands on magic link page (no auth) → enters carrier + tracking → submits → confirmation shown → optional "View dashboard" CTA.
PATH B — Portal first (Sprint 5): Roaster signs in to portal → sees "To ship" queue → taps an order → OrderFulfillmentSheet renders inside the portal → enters carrier + tracking → submits → returns to order list.
SHARED: Both paths use OrderFulfillmentSheet from packages/ui. The component prop isAuthenticated=false on the magic link page, isAuthenticated=true in the portal. Navigation context, payout breakdown behaviour, and action endpoints differ — visual design and interaction patterns are identical.
PHASE 2: EasyPost adds a third option to both paths — "Generate label" replaces the manual tracking entry. The form structure is designed for this replacement: carrier picker becomes label type picker, tracking field becomes auto-populated.

Key architecture decision — fresh links at each escalation
Each escalation tier creates a new MagicLink record. Old links are NOT invalidated.
Rationale: a roaster who opens a T+24h email after receiving the T+48h email should still be able to fulfill the order — the link works regardless of age as long as it is within its own 72h TTL and the order is still CONFIRMED.
Audit trail: each MagicLink.resend_reason field records which escalation tier created it. The history of all magic links for an order serves as the audit trail requested.
The fulfillment page loads the order by order_id from the link payload — not by token. This means any valid token for the same order reaches the same order detail.

EasyPost Phase 2 integration points
EasyPost is explicitly excluded from Sprint 4 and 5 scope. However, every story in this document calls out EasyPost integration points so the architecture is designed to accommodate it without refactoring. The key design decisions that enable this:
	•	The OrderFulfillmentSheet carrier selector is structured as a segmented control — it becomes a label-type selector when EasyPost ships. No form structure change needed.
	•	The tracking number text field will be replaced with an auto-populated read-only field when EasyPost generates the label. The form layout accommodates this.
	•	Order.shipping_label_url and Order.label_generated_by (ROASTER | PLATFORM) are already in the schema — no migration needed for Phase 2.
	•	RoasterShippingRate already has a Phase 2 comment: carrier_account_id added for EasyPost. When EasyPost ships, this field is populated.

As a roaster, I want to to click a link in my order email and see a focused, fast-loading page that shows me exactly what to ship and lets me enter tracking with minimal taps
so that I can fulfill an order in under 60 seconds without logging in to any system

Dependencies: US-10-00  ·  US-05-01
Acceptance Criteria
#
Acceptance Criteria
AC1
GET roasters.joeperks.com/fulfill/[token] — no authentication required. Token validation: exists, not expired (72h TTL), used_at IS NULL, purpose = ORDER_FULFILLMENT.
AC2
Page structure (top to bottom): (1) Joe Perks wordmark — minimal header, no nav; (2) Order summary card; (3) Ship-to address card; (4) Payout card; (5) Fulfillment form; (6) Can't fulfill link.
AC3
Order summary card: order number (JP-XXXXX) as the page headline, org name and campaign, ordered items as a clean list (product name + variant + qty + line total), order total.
AC4
Ship-to address card: formatted shipping address with copy-to-clipboard button on each line — tap to copy name, tap to copy address — for easy hand-off to shipping software.
AC5
Payout card: "Your earnings this order" in large green type ($X.XX), followed by a collapsed breakdown toggle "See breakdown" — expanding to show: product earnings / org cut / platform fee / Stripe fee / shipping (full passthrough). Breakdown is secondary — earnings headline is primary.
AC6
Fulfillment form contains exactly two fields: (1) Carrier — segmented control: UPS | FedEx | USPS | DHL | Other. On mobile, renders as a scrollable horizontal pill selector. (2) Tracking number — text input, inputMode="text", autocomplete="off", spellcheck="false". Below the input: "Tip: Copy from your carrier's app, then paste here."
AC7
Optional fulfillment note field: collapsed by default behind a "+ Add a note for the buyer" disclosure link. Expands to a single textarea (max 200 chars, character count shown). Stored in Order.fulfillment_note and included in the shipped email to buyer.
AC8
"Mark as Shipped" button: full-width, brand green, 56px tall (minimum 44px per HIG), prominent. Button text changes to a spinner while submitting. Disabled until carrier is selected AND tracking number is non-empty.
AC9
On submit success: the form is replaced with a confirmation state — checkmark icon, "Order marked as shipped", tracking info echoed back, payout summary, and a "View your dashboard" link to the authenticated portal. No page navigation.
AC10
On submit error (network failure): inline error below the button — "Something went wrong. Your order has not been marked as shipped. Try again." Retry is safe — the submission is idempotent (checks if order is already SHIPPED before processing).
AC11
Expired token (>72h): dedicated expired state page — "This fulfillment link has expired." with the order number, a "Request a new link" button (POST /api/roaster/request-new-link?order_id=X — sends a fresh magic link to the roaster's email on file), and support contact.
AC12
Already-fulfilled token: "This order has already been marked as shipped" with the tracking number and carrier echoed back, shipped timestamp, and "View your dashboard" link.
AC13
Page must load in under 2 seconds on a 4G mobile connection — server-rendered, minimal JS bundle, no client-side data fetching for the initial load.

Technical Notes
	•	Token validation uses atomic DB update: UPDATE MagicLink SET used_at=now() WHERE id=X AND used_at IS NULL — zero rows means already used. Check this before any Order mutation.
	•	Submit handler: (1) validate token, (2) check Order.status = CONFIRMED (idempotency), (3) update Order in a transaction: status=SHIPPED, shipped_at=now(), tracking_number, carrier, fulfillment_note, (4) set MagicLink.used_at, (5) logOrderEvent(ORDER_SHIPPED, ROASTER), (6) sendEmail(order_shipped → buyer). Steps 5-6 use Promise.all for parallelism.
	•	The page is a Next.js Server Component loading order data at request time using the token. No useEffect, no client-side fetch for initial render.
	•	Request-new-link endpoint: POST /api/roaster/request-new-link — validates order_id belongs to this roaster (via email on file), rate-limited to 3 requests/hour/order, creates new MagicLink with resend_reason=ADMIN_MANUAL, sends fulfillment email.
	•	"View your dashboard" link: roasters.joeperks.com/dashboard — roaster must sign in via Clerk if not already authenticated. Do not issue a session cookie from the magic link page — the magic link auth scope is single-order only.
	•	NEVER expose: buyer payment method, full card details, platform transfer amounts, or other roaster earnings in the same view. Payout breakdown is roaster-specific only.

UX / Apple HIG Notes
	•	APPLE HIG — Navigation: the page has no navigation bar, no back button, no sidebar. It is a focused task screen — like a modal sheet. The only exits are "Mark as Shipped" (primary action) and "Can't fulfill this order" (destructive secondary). All other navigation is intentionally removed.
	•	APPLE HIG — Typography hierarchy: order number is the largest text on the page (acts as the page title). Payout earnings are the second-largest. Everything else is secondary. The visual hierarchy tells the roaster: "here is your order, here is what you earn, now ship it."
	•	APPLE HIG — Touch targets: all interactive elements are minimum 44×44pt. The carrier segmented control segments are each minimum 64pt wide. The Mark as Shipped button is 56pt tall. The copy-to-clipboard targets on the address card are minimum 44pt.
	•	APPLE HIG — Feedback: every tap produces immediate visual feedback. Carrier selection highlights immediately. Submit button shows a spinner within 100ms. Success state animates in (fade + subtle scale). Error state does not shake or flash — it appears calmly below the button.
	•	APPLE HIG — Destructive actions: "Can't fulfill this order" is presented as a text link in a secondary style, not a button. It is intentionally less prominent than the primary action. It requires a second confirmation step before any action is taken.
	•	APPLE HIG — Color: green is used exclusively for the payout amount and the primary CTA. Amber/orange is used for SLA urgency. Red is used only for errors and the "Can't fulfill" confirmation destructive action. Brand green should not appear in error states.
	•	On desktop: the form is centered in a max-width 520px card on a light gray background. This creates a focused modal-like appearance without being a literal modal.
	•	On mobile: the page fills the viewport. The carrier segmented control scrolls horizontally if needed. The submit button is sticky at the bottom of the viewport on screens shorter than 700px — always visible without scrolling.
	•	Copy-to-clipboard on the address: show a "Copied!" tooltip for 1.5 seconds after tapping. Do not use a persistent icon change — the tooltip disappears and the original state returns. This follows iOS convention exactly.

Phase 2 Integration Points
	•	EasyPost integration: replace the tracking number text field with a "Generate shipping label" CTA that calls the EasyPost API. The tracking number is then auto-populated from the generated label. The current form structure (carrier + tracking field) is designed to be replaced — the carrier selector becomes the label type selector.
	•	The fulfillment_note field maps directly to EasyPost parcel notes — no schema change needed.

Definition of Done
	•	Page loads in <2s on Lighthouse mobile simulation
	•	Full fulfillment flow tested: click link → select carrier → enter tracking → submit → order status = SHIPPED in DB → buyer shipped email received
	•	Expired link shows expired state with request-new-link button
	•	Already-used link shows fulfilled state with tracking echoed
	•	Copy-to-clipboard works on Safari mobile

Acceptance Criteria
#
Acceptance Criteria
AC1
"Can't fulfill this order" link appears at the bottom of the fulfillment page (US-10-01). Tapping it opens a focused escalation sheet (full-page on mobile, modal on desktop).
AC2
Escalation sheet step 1 — Reason: required single-select. Options: "Out of stock — I don't have this product available" | "Can't meet the deadline — I need more time" | "Wrong address — the shipping address appears incorrect" | "Other issue — something else is preventing fulfillment."
AC3
Escalation sheet step 2 — Resolution offer: roaster must select what they are willing to do. Options vary by reason: Out of stock → [Request platform re-route to another roaster | Cancel and refund the buyer | Ship when restocked (needs buyer approval)]. AC4
Escalation sheet step 3 — Note: optional free-text (max 300 chars) for additional context. Placeholder: "Anything else the Joe Perks team should know?"
AC5
Submit: updates Order with flag_reason, flag_note, resolution_offered, flagged_at. Sets admin_acknowledged_flag = false. Logs OrderEvent ORDER_FLAGGED with actor=ROASTER and payload containing all three fields. Sends admin alert email.
AC6
Admin dashboard (admin.joeperks.com/orders): flagged orders show a red badge "Roaster flagged" in the order list. The flag badge persists until admin sets admin_acknowledged_flag = true via the "Acknowledge" button in order detail.
AC7
Admin order detail shows the flag reason, resolution offered, and roaster note in a prominent amber callout box at the top of the order detail view.
AC8
When a flag is submitted with any reason: the SLA timer is paused for admin review — the order is tagged as FLAGGED_PENDING_REVIEW and excluded from auto-refund calculations until admin acts. This is stored as a new status flag (admin_acknowledged_flag=false) rather than a new OrderStatus value.
AC9
After flagging, the magic link fulfillment page shows a confirmation: "We've notified the Joe Perks team. You'll hear from us within 4 hours. Your order: JP-XXXXX." No further action required from the roaster on this page.
AC10
Roaster receives a confirmation email: subject "We received your issue report for #JP-XXXXX" — confirms the platform has been notified, restates the resolution they offered, and gives a support email for follow-up.

Technical Notes
	•	SLA timer pause implementation: the sla-check Inngest job must skip orders where admin_acknowledged_flag = false AND flagged_at IS NOT NULL. Add this filter to the job query.
	•	Admin alert email: sent immediately on flag submission. Contains: order number, roaster name, flag reason (human-readable), resolution offered, roaster note, direct link to admin.joeperks.com/orders/[id].
	•	The resolution_offered options are defined as constants in packages/types — not hardcoded in the UI or API. This allows new options to be added without modifying the form component.
	•	Confirmation email to roaster uses EmailLog dedup on (order_id, "order_flagged_confirmation") — safe to call even if roaster submits twice.

UX / Apple HIG Notes
	•	APPLE HIG — Destructive actions: the escalation sheet must open with clear context at the top: "Tell us what's happening with order #JP-XXXXX." The roaster should never feel they are being punished for flagging — the copy should be supportive.
	•	APPLE HIG — Progression: the three-step sheet (Reason → Resolution → Note) uses a visual step indicator (1 of 3, 2 of 3). Back navigation is available at each step. Submit is only available on step 3.
	•	APPLE HIG — Confirmation: after submission, the confirmation screen shows a green checkmark and warm reassuring copy. The roaster's job is done. Do not leave them in an anxious state.
	•	The resolution options should use natural language, not technical terms. "Cancel and refund the buyer" not "INITIATE_REFUND". Each option should have a one-line explanation of what it means.
	•	On mobile, the escalation sheet slides up from the bottom as a full-screen sheet with a drag handle — matches the iOS sheet interaction model exactly.

Definition of Done

	•	Admin dashboard shows red "Roaster flagged" badge on the order
	•	SLA auto-refund job skips flagged-and-unacknowledged orders
	•	Admin can acknowledge flag and see all three roaster-submitted fields

#
Acceptance Criteria

AC7
Outstanding debt rows (RoasterDebt WHERE settled=false): shown in the Disputes section as "Amount owed" cards with: reason (human-readable: "Dispute fee", "Processing fee recovery", "Chargeback recovery"), amount, linked order number, settlement status.
